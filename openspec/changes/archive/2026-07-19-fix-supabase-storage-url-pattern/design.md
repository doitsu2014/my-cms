# Fix Supabase Storage URL Pattern — Design

## Context

`SupabaseStorage` in `apps/api/application_core/src/commands/media/supabase_storage.rs` historically used the legacy Supabase Storage URL shape for several object operations:

```text
/storage/v1/object/<operation>/public/{bucket}/{path}
```

Supabase Storage v1.60.2 uses the bucket as the single segment after the operation for most object endpoints:

```text
/storage/v1/object/{bucket}/{path}
/storage/v1/object/info/{bucket}/{path}
/storage/v1/object/list/{bucket}
```

The implementation predated this change. Its `download`, `get_info`, and `list_objects` URL templates retained the legacy `/public/` type segment, while `upload`, `delete`, and `delete_batch` already used the single-segment form. This created an inconsistent object-operation contract inside the same storage adapter.

The governing specification did not drift. The object-operation requirements in `openspec/changes/media-bucket-management/specs/media-bucket-management/spec.md:195-245` already require single-segment URLs, including:

- List: `/storage/v1/object/list/{bucket}` at lines 205-210.
- Upload: `/storage/v1/object/{bucket}/{path}` at lines 212-216.
- Single delete: `/storage/v1/object/{bucket}/{path}` at lines 218-222.
- Batch delete: `/storage/v1/object/{bucket}/delete` at lines 224-228.
- Metadata: `/storage/v1/object/info/{bucket}/{path}` at lines 230-234.

The bug was therefore implementation drift, not a requirements defect.

Empirical verification against the running local Supabase Storage v1.60.2 stack established the endpoint contract:

| Endpoint | Single-segment result | Legacy `/public/` result | Required behavior |
|---|---|---|---|
| `POST /object/list/{bucket}` | 200, objects returned | 400, `Bucket not found` | Single segment |
| `GET /object/{bucket}/{path}` | 200, file returned | 200, file returned | Single segment per spec |
| `GET /object/info/{bucket}/{path}` | 200, metadata returned | 200, metadata returned | Single segment per spec |
| `GET /render/image/{bucket}/{path}?width=…` | 404 | 200, transformed image returned | Keep `/public/` |

The checked-out implementation already contains the empirical production fix at `supabase_storage.rs:160-164`, `:200-204`, and `:260-264`. This document records the intended architecture and test contract so future changes do not reintroduce the legacy API-call pattern.

## Goals

- Ensure all outgoing object-operation calls use the single-segment URL pattern:
  - `upload`
  - `download`
  - `get_info`
  - `list_objects`
  - `delete`
  - `delete_batch`
- Preserve the legacy `/storage/v1/object/public/{bucket}/{path}` shape generated by `public_url()` for client-facing public object URLs; it is not an internal authenticated API-call URL.
- Preserve `/storage/v1/render/image/public/{bucket}/{path}` for image rendering because Supabase Storage v1.60.2 still requires the `/public/` segment.
- Make Wiremock tests assert the single-segment API paths so a regression fails in CI.
- Add a focused URL contract test that documents the normal object-operation pattern and the two intentional client-facing exceptions.
- Leave the existing specification unchanged.

## Non-Goals

- Changing the public, browser-facing URL shape produced by `public_url()`.
- Changing the image-render endpoint to a single-segment URL.
- Changing media API response models, authentication, error mapping, or bucket-selection behavior.
- Refactoring `SupabaseStorage` beyond the URL templates and a version-dependency comment for image rendering.
- Introducing a generalized URL-builder abstraction.
- Changing Supabase configuration, environment variables, database schema, or stored data.

## Decisions

### 1. Use single-segment URLs for object operations except render-image

**Decision:** Every authenticated outgoing object-operation request SHALL place the bucket directly after the object operation:

```text
POST   /storage/v1/object/{bucket}/{path}
GET    /storage/v1/object/{bucket}/{path}
GET    /storage/v1/object/info/{bucket}/{path}
POST   /storage/v1/object/list/{bucket}
DELETE /storage/v1/object/{bucket}/{path}
DELETE /storage/v1/object/{bucket}/delete
```

The three drifted methods change as follows:

- `download`: `/object/public/{bucket}/{path}` becomes `/object/{bucket}/{path}`.
- `get_info`: `/object/info/public/{bucket}/{path}` becomes `/object/info/{bucket}/{path}`.
- `list_objects`: `/object/list/public/{bucket}` becomes `/object/list/{bucket}`.

`render_image_url()` remains:

```text
/storage/v1/render/image/public/{bucket}/{path}
```

**Rationale:** This matches the existing OpenSpec requirements and the empirically verified Supabase Storage v1.60.2 API. Render-image is a confirmed exception: removing `/public/` causes a 404 in v1.60.2.

Add a comment near `render_image_url()` at `supabase_storage.rs:91-95` explaining that the legacy segment is intentional and version-dependent.

### 2. Split `public_url()` semantics from authenticated API-call URLs

**Decision:** `public_url(path)` at `supabase_storage.rs:84-89` retains:

```text
{SUPABASE_URL}/storage/v1/object/public/{bucket}/{path}
```

Its semantic role is a client-facing, browser-readable public object URL. It SHALL NOT be reused to construct authenticated internal Storage API requests.

`download()` constructs its internal URL directly at `supabase_storage.rs:161-164`:

```rust
format!(
    "{}/storage/v1/object/{}/{}",
    self.supabase_url, self.bucket, path
)
```

The request continues to include Bearer and `apikey` authentication at `supabase_storage.rs:165-170`.

**Rationale:** A client-facing public URL and an authenticated Storage API URL do not have the same path contract. Reusing `public_url()` inside `download()` conflated these responsibilities and reintroduced `/public/` into an internal API call. Keeping the helper while constructing the authenticated URL explicitly makes the distinction visible without adding a new abstraction.

### 3. Update all 11 affected Wiremock path assertions

**Decision:** The eleven tests that exercise the three drifted methods SHALL expect single-segment paths.

| Test function | New expected path |
|---|---|
| `download_returns_bytes_and_content_type_on_200` | `/storage/v1/object/media/foo.png` |
| `download_returns_not_found_on_404` | `/storage/v1/object/media/missing.png` |
| `download_returns_not_found_on_supabase_400_with_statuscode_404` | `/storage/v1/object/xx/foo.png` |
| `download_returns_storage_error_on_500` | `/storage/v1/object/media/oops.png` |
| `get_info_parses_json_response` | `/storage/v1/object/info/media/foo.png` |
| `get_info_returns_not_found_on_404` | `/storage/v1/object/info/media/missing.png` |
| `get_info_returns_not_found_on_supabase_400_with_statuscode_404` | `/storage/v1/object/info/xx/foo.png` |
| `list_objects_posts_with_correct_body_and_returns_list` | `/storage/v1/object/list/media` |
| `list_objects_returns_not_found_on_supabase_400_with_statuscode_404` | `/storage/v1/object/list/xx` |
| `list_objects_returns_not_found_on_does_not_exist_body` | `/storage/v1/object/list/xx` |
| `list_objects_returns_not_found_on_http_404` | `/storage/v1/object/list/xx` |

These assertions currently appear at `supabase_storage.rs:867`, `:889`, `:904`, `:920`, `:951`, `:968`, `:983`, `:1024`, `:1094`, `:1111`, and `:1127`.

Add one focused asynchronous contract test named `url_patterns_match_spec_single_segment`. It should use a non-default bucket and Wiremock expectations to pin the corrected `download`, `get_info`, and `list_objects` paths. The test should also assert that `public_url()` and `render_image_url()` retain their intentional `/public/` segments.

**Rationale:** The previous mocks mirrored the implementation bug, allowing the incorrect paths to pass CI. The corrected assertions make the tests act as an external API contract. The focused contract test records the distinction between internal object calls and client-facing URL builders in one place.

### 4. Make no specification changes

**Decision:** Do not modify `openspec/changes/media-bucket-management/specs/media-bucket-management/spec.md`.

**Rationale:** The spec already requires the correct single-segment object-operation URLs. Only implementation and tests drifted. Syncing or archiving this corrective change must not alter the capability requirements.

## Risks / Trade-offs

| Risk | Impact | Mitigation |
|---|---|---|
| Downstream code relies on the legacy API-call path | Object reads, metadata, or listing could fail after the path changes | Search confirmed that `create/create_handler.rs:45-63`, `list/list_handler.rs:34-50`, and `read/metadata_handler.rs:23-39` construct response URLs after operations; they do not construct the outgoing `SupabaseStorage` API-call paths. |
| `public_url()` is reused for an internal authenticated request | The `/public/` segment could be reintroduced | Document its client-facing semantics, keep `download()` URL construction separate, and pin both patterns in tests. |
| A future Supabase version changes render routing | Image transformations could begin returning 404 | Keep the empirically required `/public/` segment and add a version-dependency comment beside `render_image_url()`. Re-test when upgrading storage-api. |
| Inline URL templates can drift independently | Methods could diverge again | Keep explicit Wiremock path assertions for each operation and add the consolidated contract test. |
| The legacy and single-segment download URLs both currently work | A developer may conclude either is acceptable | Prefer the specification and v1.60.2 primary API contract; tests intentionally reject the legacy internal path. |

The deliberate trade-off is retaining explicit format strings rather than introducing a URL-builder abstraction. Only three templates were wrong, and refactoring the adapter is outside this corrective change.

## Migration Plan

No data migration, database migration, environment change, API response schema change, or specification change is required.

Code-only rollout:

1. Update the three production URL constructions in `download`, `get_info`, and `list_objects`.
2. Add the render-image version-dependency comment.
3. Update the 11 Wiremock path assertions.
4. Add the focused URL contract test.
5. Run the complete verification gate.
6. Deploy the API normally.

No coordinated frontend deployment is required.

**Rollback:** Revert the three production format-string changes and their corresponding test assertions. This restores the previous behavior but also reintroduces the v1.60.2 list failure and the implementation/spec mismatch. There is no data or configuration rollback.
