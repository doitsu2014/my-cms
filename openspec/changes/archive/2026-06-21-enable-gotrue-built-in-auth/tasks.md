## 6. NEW: repair Kong gateway so authenticated Supabase API calls work

**Discovered during implementation of tasks 3.3 and 4.x.** The Kong config at `volumes/api/kong.yml` enabled the `key-auth` plugin on `auth-v1`, `rest`, `rest-graphql`, `realtime`, `meta`, and `analytics` services, but the file was **missing the top-level `consumers:` section**. Without registered consumers, `key-auth` rejected every request that was not on the `*-open*` services (the `/auth/v1/verify`, `/auth/v1/callback`, `/auth/v1/authorize`, `/auth/v1/sso/saml` routes, which intentionally have no auth). This was a pre-existing bug in the local Supabase stack, and it is **the root cause** of the `400 "Unsupported provider: provider is not enabled"` error reported in the original problem: GoTrue was never reached because Kong rejected the `apikey: devkey` call.

A second pre-existing bug surfaced during this work: `ANON_KEY` and `SERVICE_ROLE_KEY` in `.env.supabase.example` / `.env.my-cms.example` were the literal string `devkey`, but GoTrue validates them as HS256 JWTs signed with `JWT_SECRET`. A non-JWT `apikey` is always rejected as `bad_jwt`. Fix: generate proper JWTs with `scripts/generate-jwt.sh` and put them in the `.env.*.example` files so a fresh checkout has a working stack.
