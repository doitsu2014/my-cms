## Why

The public blog reader currently lives in a sibling repository (`my-blogs-rsbuild/client_side`) as a React 19 + Rsbuild + Express 5 SSR application. It hardcodes production URLs (`https://my-cms-api.ducth.dev/graphql/immutable`, `https://my-cms-api.ducth.dev/media/...`, `https://my-blogs.ducth.dev`, the avatar media path) and has its own Docker image, its own Helm chart, and its own GitHub release pipeline. The sibling repo is therefore the only place where this codebase is tested, containerised, and shipped. Moving it into `my-cms` under the product identity `ducth-dev-website` keeps the reader adjacent to the headless CMS it talks to, removes URL hardcoding by sharing the same env-backed runtime-configuration pattern already proven by `apps/web` (see `apps/web/src/config/runtime-config.ts:32-57` and `apps/web/entrypoint.sh:9-31`), and lets a single Docker Swarm stack (`deployments/docker-swarm/`) host the admin, the API, and the reader without a parallel deployment pipeline.

## What Changes

- Add a new standalone app at `apps/ducth-dev-website/` mirroring the `apps/web/` layout (own `package.json`, `pnpm-workspace.yaml`, `pnpm-lock.yaml`, `tsconfig.json`, `eslint.config.mjs`, `postcss.config.mjs`, `tailwind.config.ts`, `.env.example`, `.dockerignore`, `.gitignore`). The pnpm package name SHALL be `ducth-dev-website`. **No top-level pnpm workspace** is introduced; each app keeps its own lockfile as today.
- Move the SSR source from `my-blogs-rsbuild/client_side/` (`App.tsx`, `AppContent.tsx`, `index.client.tsx`, `index.server.tsx`, `pages/*`, `components/layout/*`, `i18n/*`, `infrastructure/graphql/*`, `config/*`) into `apps/ducth-dev-website/src/`. The Rsbuild config SHALL keep the two existing environments (`web` for the client bundle, `node` for the SSR bundle) so `dist/client/*` and `dist/server/index.mjs` are both produced. **No new visual redesign.** Routing, layouts, pages, i18n, and GraphQL queries are preserved verbatim.
- Replace every hardcoded production URL or brand string in `src/config/api.config.ts`, `src/config/site.config.ts`, `src/pages/HomePage.tsx`, `src/pages/CategoriesPage.tsx`, `src/pages/CategoryDetailPage.tsx`, `src/pages/PostDetailPage.tsx`, and `index.html` with values resolved from a single source of truth: the Express server's `process.env`. The server injects the resolved values into the SSR HTML as a single `<script>` tag with a stable id, and the client bundle reads from `window.__APP_CONFIG__` exactly as `apps/web/src/config/runtime-config.ts:32-57` does today.
- Add `Dockerfile` (three-stage `node:20-alpine`: deps → build → non-root Express runner), `nginx.conf` is **not** added (the runner is the Express server itself). HEALTHCHECK on `/en`, `EXPOSE 3001`, non-root user `expressjs:1001`. Pattern mirrors `my-blogs-rsbuild/client_side/Dockerfile:1-57`.
- Add `deployments/docker-swarm/apps/docker-compose.yaml` service `ducth-dev-website` (container name `ducth-dev-website`, `depends_on: my-cms-api { condition: service_started }`, joins `supabase_network`, exposes port 3001 only to Traefik via labels). The env contract is a single `WEBSITE_*` namespace consumed by the service (`WEBSITE_SITE_NAME`, `WEBSITE_SITE_URL`, `WEBSITE_AVATAR_URL`, `WEBSITE_DEFAULT_TITLE`, `WEBSITE_DEFAULT_DESCRIPTION`, `WEBSITE_PUBLIC_GRAPHQL_API_URL`, `WEBSITE_PUBLIC_GRAPHQL_CACHE_API_URL`, `WEBSITE_PUBLIC_MEDIA_BASE_URL`) plus `NODE_ENV=production` and `PORT=3001`.
- Add a Traefik router `website` and service `website-service` in `deployments/docker-swarm/traefik/dynamic/my-cms.yml` routing `Host(`ducth.dev`)` to `http://ducth-dev-website:3001`. Extend the existing `api-cors` middleware's `accessControlAllowOriginList` with `https://ducth.dev`. Add `CMS_WEBSITE_HOST=ducth.dev` to `deployments/docker-swarm/traefik/.env.example` and to the apps compose's `depends_on`/label section. The Compose labels and the static dynamic file are both edited, but the routing truth is the static dynamic config; Compose labels add only the gzip middleware.
- Document the env contract in `apps/ducth-dev-website/.env.example` (build-time) and in `deployments/docker-swarm/apps/.env.example` (runtime), mirroring `apps/web/.env.example:1-48` and `deployments/docker-swarm/apps/.env.example:75-85`.
- Update `deployments/docker-swarm/README.md` to include the website service in the access-points table.
- Validate required URLs at startup: if `WEBSITE_PUBLIC_GRAPHQL_API_URL`, `WEBSITE_PUBLIC_MEDIA_BASE_URL`, or `WEBSITE_SITE_URL` is missing, the Express server exits non-zero with a log line that names the missing variable. Sanitise every error logged by the SSR handler so stack traces and request bodies are never written to logs.

## Capabilities

### New Capabilities

- `website-frontend`: Public reader SSR client behavior — locale-prefixed routing, SSR + hydration contract, runtime configuration contract (server source of truth, HTML injection, browser access, required values, fallback policy, safe serialization), public GraphQL queries (`/graphql/immutable`), media URL resolution.
- `website-deployment`: Production containerization and stack wiring — three-stage Node 20 Alpine Dockerfile, Docker Compose service contract, Traefik Host-based routing, env contract, healthcheck, independent rollback.

### Modified Capabilities

- `local-dev-environment`: The apps compose SHALL run an additional `ducth-dev-website` service on the shared `supabase_network`, depending on `my-cms-api` having started, so the reader joins the same local stack as the admin and the API.

## Impact

- **Code**: New directory `apps/ducth-dev-website/` (~30 files). No edits to `apps/web/**`, `apps/api/**`, `apps/web/Dockerfile`, `apps/web/nginx.conf`, `apps/web/entrypoint.sh`.
- **Deployment**: `deployments/docker-swarm/apps/docker-compose.yaml` (add one service), `deployments/docker-swarm/apps/.env.example` (add `WEBSITE_*` and `CMS_WEBSITE_HOST`), `deployments/docker-swarm/traefik/dynamic/my-cms.yml` (add router + service + extend CORS allow-list), `deployments/docker-swarm/traefik/.env.example` (add `CMS_WEBSITE_HOST`), `deployments/docker-swarm/README.md` (update access-points table). The Supabase stack (`deployments/docker-swarm/supabase/`) is **not** touched.
- **Dependencies**: `react@^19.1.1`, `react-dom@^19.1.1`, `@apollo/client@^3.14.0`, `react-router-dom@^7.11.0`, `react-i18next@^16.5.1`, `i18next@^25.7.4`, `daisyui@^5.5.14`, `tailwindcss@^4.1.18`, `express@^5.2.1`, `express-rate-limit@^8.2.1`, `highlight.js@^11.11.1`, `graphql@^16.12.0` (all migrated from `my-blogs-rsbuild/client_side/package.json:15-32`). Dev deps migrated verbatim. The existing `my-blogs-rsbuild/client_side` repo remains the source of truth until the migration is verified end-to-end; **this change does not delete that repo**.
- **APIs consumed**: `GET /graphql/immutable` (public GraphQL, no auth — see `apps/api/src/bin/my-cms-api.rs:108-113`), `GET /media/{*path}` and `GET /media/images/{*path}` (public media — see `apps/api/src/bin/my-cms-api.rs:99-107`). No backend changes.
- **Network**: New container joins the existing external `supabase_network` (`deployments/docker-swarm/apps/docker-compose.yaml:172-175`). The new container can reach `my-cms-api:8989` by DNS on that network.
- **CI**: This change does **not** add a release-image workflow or publish to a registry. Local `pnpm lint`/`typecheck`/`build` and Docker/Compose smoke tests are run as verification gates.
