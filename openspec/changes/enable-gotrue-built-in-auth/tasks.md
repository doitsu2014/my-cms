## 6. NEW: repair Kong gateway so authenticated Supabase calls work

**Discovered during implementation of tasks 3.3 and 4.x.** The Kong config at `volumes/api/kong.yml` enables the `key-auth` plugin on `auth-v1`, `rest`, `rest-graphql`, `realtime`, `meta`, and `analytics` services (lines 44–60, 61–78, 79–96, 97–114, 124–139, 140–155), but the file is **missing the top-level `consumers:` section**. Without registered consumers, `key-auth` rejects every request that is not on the `*-open*` services (the `/auth/v1/verify`, `/auth/v1/callback`, `/auth/v1/authorize`, `/auth/v1/sso/saml` routes, which intentionally have no auth). This is a pre-existing bug in the local Supabase stack (the stack has been up for 6 hours, so the bug was there before this change started), and it is **the root cause** of the `400 "Unsupported provider: provider is not enabled"` error reported in the original problem: GoTrue was never reached because Kong rejected the `apikey: devkey` call.

Upstream Supabase's `volumes/api/kong.yml` has a `consumers` block plus a `request-transformer` plugin (which rewrites the `Authorization` header into a GoTrue-mintable JWT per request). The local copy is a stripped-down variant that omits both — see the comment on `docker-compose.supabase.yaml:384-387` which says "kong:2.8.1 reads KONG_DECLARATIVE_CONFIG directly. Our volumes/api/kong.yml contains zero $VAR references, so no env-substitution entrypoint is needed." That is true for the YAML body but incorrect as a claim of "fully working": Kong itself still requires consumers for `key-auth` to authorize anything.
